# 仓库地图

## 1. 目录职责

| 路径 | 类型 | 作用 | 是否直接编辑 |
| --- | --- | --- | --- |
| `CMakeLists.txt` | 构建入口 | 定义项目名、语言标准、目标文件 | 是 |
| `src/` | 源码目录 | 放置项目源文件 | 是 |
| `src/main.cpp` | 程序入口 | 当前唯一源码入口 | 是 |
| `rei/rule/` | 规则目录 | 维护仓库级规则与文档约束 | 是 |
| `build/` | 生成目录 | CMake 生成文件、缓存、可执行产物 | 否 |

## 2. 当前最小结构认知

当前仓库还不是一个复杂的多模块工程，而是一个最小可构建的 C++20 项目。任何规则都必须围绕这个事实编写。

这意味着：

- 不要假定存在库目录、测试目录、平台抽象层或插件系统。
- 不要写出“先读某个模块总文档”之类当前无法执行的要求。
- 新增源文件时，要同步更新 `CMakeLists.txt`。

## 3. 常见任务落点

| 任务 | 首选修改位置 | 说明 |
| --- | --- | --- |
| 修改语言标准 | `CMakeLists.txt` | 以 `CMAKE_CXX_STANDARD` 为准 |
| 修改程序入口逻辑 | `src/main.cpp` | 当前程序唯一入口 |
| 新增源文件 | `src/` + `CMakeLists.txt` | 新文件必须纳入构建 |
| 修改仓库规则 | `rei/rule/*.md` | 先改规则，再改其他说明 |
| 清理构建产物 | `build/` | 只处理生成内容，不改源码 |

## 4. 禁止事项

- 不要直接手改 `build/` 下的生成文件。
- 不要把 `build/venthmi` 当成源码或配置入口。
- 不要在规则文档里引用当前不存在的目录。
- 不要把样例性的建议写成强制事实。

## 5. 扩展项目时的同步要求

如果仓库从最小项目扩展为多文件或多模块结构，至少同步更新下面内容：

1. `map.md` 中的目录职责表。
2. `projectrule.md` 中的构建与目录规则。
3. `coderule.md` 中新增目录对应的命名与实现约束。
