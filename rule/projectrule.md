# 项目规则

## 1. 项目基线

- 构建系统：CMake。
- 语言：C++20。
- 当前目标：单个可执行程序 `venthmi`。
- 当前入口：`src/main.cpp`。
- 当前构建输出位于 `build/`。

项目规则必须服务于“最小、清晰、可持续扩展”的目标，避免过早引入复杂分层。

## 2. 构建规则

- `CMakeLists.txt` 是唯一权威构建入口。
- 语言标准必须显式设置为 C++20，并保持 `CMAKE_CXX_STANDARD_REQUIRED ON`。
- 新增 `.cpp` 文件后，必须同步更新目标的源文件列表。
- 非必要情况下，不要把编译选项拆到多个分散文件。
- 不要修改 `build/` 中的 `Makefile`、`CMakeCache.txt` 或 `CMakeFiles/` 内容来“修复”构建问题；应回到 `CMakeLists.txt` 修正。

## 3. 目录规则

- `src/` 只放源代码。
- `rei/rule/` 只放规则文档，不放业务说明或临时草稿。
- `build/` 只存放生成产物。
- 未来如果增加 `include/`、`tests/` 或 `docs/`，要先在 `map.md` 中登记，再开始大量使用。

## 4. 变更边界

- 功能修改应优先落在源码目录，而不是通过更改生成产物绕过去。
- 构建问题优先检查 `CMakeLists.txt` 与源码是否一致。
- 规则更新必须基于当前仓库真实状态，不为未落地的架构预留强制规范。
- 如果只是修复一个行为问题，不要顺手做大范围结构重写。

## 5. 文档同步规则

出现下面任一变化时，应同步更新规则文档：

- C++ 标准变更。
- 可执行目标名称变更。
- 新增一级源码目录。
- 引入测试框架或新增测试目录。
- 构建方式从单目标变为多目标。

## 6. 当前阶段推荐做法

- 保持程序入口简单明确。
- 优先使用标准库能力解决问题。
- 只有在真实需求出现时，再拆分类、模块和子目录。
- 扩展工程时先做最小可运行改动，再补结构优化。
