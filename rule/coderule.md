# C++20 编码规则

## 1. 文本与基础格式

- 开发环境以 VS Code + Linux 为当前基线。
- 文本编码统一为 UTF-8。
- 缩进统一为 4 个空格。
- 换行统一为 LF。
- 每个文本文件以且仅以一个换行符结尾。
- 文件名和目录名保持小写；如需分词，优先使用下划线。
- 规则文档默认使用中文；代码标识符使用英文。

## 2. 通用实现原则

- 先写清晰、可读、可构建的实现，再考虑抽象。
- 除非需求明确，否则不要把重构和行为变更混在一起。
- 修改代码时优先修根因，不做表面绕行。
- 非必要不要引入第三方库；优先使用标准库。
- 新增文件后必须让构建系统感知到该文件。

## 3. 命名规则

- 类型名使用 `PascalCase`，例如 `AppConfig`。
- 函数名使用 `camelCase`，例如 `loadConfig`。
- 局部变量使用 `camelCase`，例如 `inputPath`。
- 常量优先使用 `constexpr`，命名使用 `kCamelCase`，例如 `kDefaultPort`。
- 命名空间使用小写，例如 `venthmi`。
- 枚举优先使用 `enum class`，枚举类型名使用 `PascalCase`。
- 避免含义空泛的命名，如 `doThing`、`tmpValue`、`commonUtil`。

## 4. 文件与结构规则

- 一个源文件只承载一组紧密相关的职责。
- 头文件只暴露必要接口，避免泄漏实现细节。
- `#include` 只保留实际使用的头文件。
- 如果某段逻辑还没有复用价值，不要过早拆成独立抽象层。
- 当源码规模扩大后，再按职责拆分为多个 `.hpp` / `.cpp` 文件。

## 5. C++20 语言使用规则

- 优先使用 C++20 标准库能力，而不是旧式替代方案。
- 能用 `constexpr`、`const`、`std::string_view`、`std::span` 时优先考虑，但前提是可读性更好。
- 资源管理优先依赖 RAII。
- 非共享所有权场景不要使用裸 `new` / `delete`。
- 只有在确实表达所有权语义时才使用智能指针。
- 布尔条件保持直接，不写晦涩的双重否定。
- 能使用 `enum class` 时不要退回传统枚举。
- 不为炫技而使用模板、概念或元编程。

## 6. 函数编写规则

- 函数保持短小，单一职责。
- 参数优先表达真实语义，不堆砌无意义包装。
- 输入只读的大对象优先使用 `const T&` 或 `std::string_view`。
- 返回值优先直接返回结果对象，而不是通过输出参数回填。
- 只有在语义稳定且确实不会抛出异常时才标记 `noexcept`。
- 发现错误时要显式处理，不能静默吞掉失败路径。

## 7. 注释与文档规则

- 注释只解释意图、约束或不明显的原因，不解释显而易见的语句。
- 不写“把值赋给变量”这类低信息注释。
- 复杂分支、边界条件和临时权衡可以用一两句简短注释说明。
- 规则文档必须基于当前仓库真实结构，不引用不存在的模块。

## 8. 样式基线

大括号使用同行风格：

```cpp
int main() {
    return 0;
}

if (ready) {
    run();
}
```
